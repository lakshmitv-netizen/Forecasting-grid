Analysis: Will This Solve the Editing Problem?

✅ Advantages:

  • Better Organization: Related measures (Quantity & Revenue) are visually grouped together, making the structure clearer.
  • Logical Grouping: Users can collapse/expand entire measure groups (e.g., all Sales Agreement measures) at once.
  • Reduced Visual Clutter: Parent rows provide a way to organize many measures without overwhelming the interface.

❌ Potential Issues:

  • Parent Row Values: What value should parent measure rows display? They can't meaningfully sum Quantity + Revenue (different units). Options:
    • Show nothing (empty cells) - but then why have the row?
    • Show count of child measures - not very useful
    • Show a placeholder like "—" - acceptable but not ideal
  • Editing Parent Rows: Should parent measure rows be editable? If yes:
    • How do you edit a "Sales Agreement" value when it contains both Quantity and Revenue?
    • Would editing propagate to both child measures proportionally?
    • This creates ambiguity and potential user confusion
  • Value Propagation Complexity: Current system propagates values upward (child → parent). With parent measures:
    • Editing "TRN 750 - A" under "Sales Agreement Quantity" → propagates to account → category → measure
    • But should it also propagate to the parent "Sales Agreement" row? What value?
    • This adds an extra layer of complexity
  • Cross-Measure Dependencies: Current logic (e.g., Planned Volume → Revenue) works at the measure level. With parent groups:
    • Dependencies still work, but the parent row adds no value to this logic
    • It's just an organizational layer

💡 Recommendation:

Parent measure rows should be READ-ONLY organizational/visual elements only. They serve to group related measures but should NOT be editable themselves. This solves the editing problem by:

1. Maintaining clarity: Users edit at the actual measure level (Quantity, Revenue), not at ambiguous parent levels.

2. Preserving existing logic: Value propagation and cross-measure dependencies continue to work exactly as they do now, just with an extra visual grouping layer.

3. Avoiding confusion: No ambiguity about what editing a parent row means, since it's not editable.

Conclusion: Yes, this structure CAN solve editing problems IF parent measure rows are treated as read-only organizational elements. The editing happens at the individual measure level (where it makes sense), and the parent rows just provide better visual organization.
